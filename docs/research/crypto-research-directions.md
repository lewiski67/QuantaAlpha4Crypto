# Crypto Research Direction Log

This document records crypto factor research directions that have already been tested or explicitly queued. Check this log before choosing a new `research_direction` so we do not repeatedly spend LLM and evaluation budget on the same mechanism.

## How to Record a Direction

Use one row per distinct market mechanism. If a later run only changes symbols, dates, frequency, or prompt wording while testing the same mechanism, update the existing row instead of adding a duplicate.

| Field | Meaning |
| --- | --- |
| Direction ID | Stable short name for the mechanism. |
| Status | `queued`, `tested`, `promising`, `rejected`, or `retired`. |
| Research mechanism | One falsifiable mechanism, not a broad theme. |
| Venue/product | Binance spot or Binance USD-M futures. |
| Tradable/evaluated universe | Symbols that may be scored for PnL and reports. |
| Feature universe and fields | Symbols/data sources and exact Crypto Panel fields the factor may read. |
| Candidate timing grid | Input lookback, update frequency, rebalance frequency, and holding/candidate horizon candidates. |
| Timing rationale | Why those windows match the mechanism. |
| Construction and avoid rules | Preferred normalizations/interactions/regime filters and prohibited fields/operations. |
| Result | Accepted count, best factor, best metrics, or reason for rejection. |
| Artifact | Workspace/report path when available. |
| Next action | Whether to rerun, mutate, narrow, broaden, or avoid. |
